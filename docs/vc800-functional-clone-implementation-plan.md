# PE/VC 投资管理系统实施计划

## 1. 推荐技术方案

### 前端

- React + TypeScript + Ant Design Pro，或 Vue 3 + TypeScript + Element Plus。
- 表格、表单、看板、流程图需要组件化封装。
- 大型表单使用 schema-driven form，但核心业务对象保留强类型模型。

### 后端

任选一条主线：

- Java/Spring Boot：适合传统企业私有化、审计、权限、流程稳定性要求高的客户。
- Node.js/NestJS：适合快速迭代、AI 集成、前后端 TypeScript 一体化。

建议服务边界：

- Auth/权限服务。
- 项目服务。
- 基金服务。
- 投资组合/现金流服务。
- 流程服务。
- 文档服务。
- 通知服务。
- AI 解析服务。
- 报表服务。

### 数据与基础设施

- PostgreSQL：主业务库。
- Redis：会话、队列、缓存。
- S3/MinIO/OSS：文件存储。
- OpenSearch/Elasticsearch：文档全文检索，V1 后引入。
- Temporal/Camunda/Flowable：流程引擎，MVP 可先做轻量内置审批。
- Docker Compose 起步，Kubernetes 用于企业版部署。

## 2. 安全基线

必须从第一天落实：

- 服务端密钥不进入前端包、不进入 HTML。
- 所有 Secret 进密钥管理或环境变量。
- Cookie 使用 `HttpOnly`、`Secure`、`SameSite=Lax/Strict`。
- API 鉴权统一中间件。
- 文件下载走授权校验和短期签名 URL。
- 导出、下载、批量操作必须审计。
- 数据按租户、组织、角色、负责人隔离。
- 生产日志脱敏，禁止记录密码、Token、密钥、完整身份证/手机号等敏感信息。

## 3. 里程碑

### 第 0 阶段：需求冻结与原型，1-2 周

产出：

- 模块范围确认。
- MVP 原型。
- 数据模型评审。
- 权限模型评审。
- 第一批流程模板确认。

验收：

- 业务方确认 MVP 不追求完整功能，只先覆盖项目、基金、投资组合、流程、文档闭环。

### 第 1 阶段：平台底座，2-3 周

范围：

- 登录、用户、组织、角色、权限。
- 审计日志。
- 通用列表、通用表单、导入导出基础能力。
- 文件上传、预览、下载。
- 通知/待办框架。

验收：

- 可创建用户和角色。
- 不同角色看到不同菜单和数据范围。
- 文件可上传并按权限访问。

### 第 2 阶段：项目管理 MVP，3-4 周

范围：

- 项目池列表。
- 项目看板。
- 项目新增/编辑。
- 项目详情。
- 项目日程、文档、会议纪要。
- 阶段变更历史。

验收：

- 投资经理能录入项目、推进阶段、上传 BP、创建会议纪要。
- 管理层能按阶段查看项目储备。
- 项目详情能看到关联文件、会议、日程。

### 第 3 阶段：基金管理 MVP，4-5 周

范围：

- 基金列表。
- 基金新增/编辑。
- 基金详情。
- 投资信息/投资组合。
- 权益变动。
- 基础现金流。

验收：

- 基金可维护基础信息、期限、规模、管理费、收益分配等关键字段。
- 基金能关联项目投资记录。
- 项目和基金两侧都能看到同一条投资关系。

### 第 4 阶段：流程、风险、投后，3-4 周

范围：

- 固定流程模板：立项、投决、用印、付款、合同审批。
- 流程中心：待办、已办、我的申请、抄送。
- 风险关键条款和提醒。
- 投后数据收集基础版。

验收：

- 用户能从项目/基金发起流程。
- 审批任务进入待办。
- 风险条款到期产生提醒。
- 投后数据可按项目记录状态。

### 第 5 阶段：报表与交付加固，2-3 周

范围：

- 首页驾驶舱。
- 项目阶段分布、基金规模、累计投资、回款、退出、风险提醒。
- 导入导出完善。
- 权限穿透测试。
- 性能和安全测试。

验收：

- 管理层工作台可查看核心指标。
- 全系统通过越权测试。
- 常用列表和详情达到可接受性能。

## 4. MVP 排期估算

适合 6-8 人小队：

- 产品经理 1
- 前端 2
- 后端 2-3
- 测试 1
- UI/交互 0.5-1
- 架构/DevOps 0.5

MVP 总工期：12-16 周。

企业级 V1：再增加 8-12 周。

- 导入导出。

- AI 会议纪要解析。
- AI BP/合同字段提取。
- 外部填报门户。

- LP 门户。

## 7. API 设计原则

不要沿用目标站接口路径。建议 REST + 少量批处理接口：

- `POST /auth/login`
- `GET /workbench/summary`
- `GET /projects`
- `POST /projects`
- `GET /projects/{id}`
- `PATCH /projects/{id}`
- `POST /projects/{id}/stage`
- `GET /funds`
- `POST /funds`
- `GET /funds/{id}`
- `GET /investments`
- `POST /investments`
- `GET /documents`
- `POST /documents/upload`
- `POST /workflows`
- `POST /workflows/{id}/tasks/{taskId}/approve`
- `GET /audit-logs`

接口规范：

- 所有列表支持 `page`、`pageSize`、`sort`、`filters`。
- 所有写接口返回对象版本号。
- 所有导出异步化，返回任务 ID。
- 所有文件接口二次鉴权。

## 8. 测试策略

单元测试：

- 权限规则。
- 金额、比例、日期校验。
- 阶段流转。
- 现金流汇总。

集成测试：

- 项目创建到投资记录闭环。
- 基金创建到投资组合闭环。
- 流程发起到审批闭环。
- 文件上传到权限下载闭环。

安全测试：

- 横向越权。
- 纵向越权。
- 导出越权。
- 文件下载越权。
- 前端密钥扫描。
- 审计完整性。

## 9. 风险与建议

### 风险 1：范围过大

PE/VC 系统天然覆盖项目、基金、流程、财务、文档、报表。建议 MVP 先把项目、基金、投资关系、流程、文档跑通，不要第一版追求全量配置化。

### 风险 2：过早做低代码

目标产品有很强的配置化痕迹。复刻时不要一开始就做完整低代码平台，否则交付会失控。建议核心对象强模型，扩展字段配置化。

### 风险 3：权限复杂

投资系统最容易出问题的是数据越权和文件越权。权限模型必须先设计，不能后补。

### 风险 4：AI 结果不可直接入库

会议纪要、BP、合同条款解析可以提升效率，但必须人工确认后写入正式字段。

### 风险 5：安全反面案例

观察中发现目标页面存在前端敏感配置暴露风险。新系统必须把密钥管理作为架构红线。

## 10. 下一步执行清单

1. 业务方确认 MVP 模块和角色。
2. 设计项目、基金、投资关系、现金流四张核心业务流转图。
3. 出低保真原型：项目看板、项目详情、基金新增、基金详情、流程审批。
4. 建库并实现 Auth/RBAC/Audit。
5. 实现项目和基金主链路。
6. 补流程、文档、风险、报表。
7. 做权限和安全专项测试。
